# 项目交接清单与第一周任务表

## 1. 交接清单

### 1.1 基础入口

- [ ] 知道当前目录是以知识底座为主的整理工作区，同时包含 `frontend/`、`bladex-boot/` 工程目录
- [ ] 读过 `README.md`
- [ ] 读过 `REPO_WIKI.md`
- [ ] 读过 `AGENTS.md`

### 1.2 外部依赖

- [ ] 补齐 `docs/00-meta/外部依赖与交接说明.md` 中的代码仓地址
- [ ] 补齐环境获取方式
- [ ] 补齐 PRD 权威版本锚点
- [ ] 确认谁维护这些信息

### 1.3 规格流转

- [ ] 知道 `changes/2026-04/` 当前只放 proposal 镜像
- [ ] 知道当前仓库的 spec / design / tasks / test-cases / acceptance / contracts 主要位于 `openspec/changes/<feature>/`
- [ ] 知道 `specs/` 是 starter 目标态目录，当前仓库未整体落盘
- [ ] 知道没有规格不直接开始实现

## 2. 第一周任务表

| 天 | 任务 | 产出 |
|---|---|---|
| Day 1 | 确认负责人、环境、外部 PRD 和代码仓 | 补齐 `docs/00-meta/外部依赖与交接说明.md` 初稿 |
| Day 2 | 阅读 overview / architecture / data-model / business-rules | 形成共识，不再口头对规则 |
| Day 3 | 检查 active feature 的 proposal | 明确范围和非范围 |
| Day 4 | 检查 `openspec/changes/<feature>/` 下的 specs、contracts、test-cases | 冻结接口和测试 |
| Day 5 | 决定是否进入实现 | 满足条件才开始写代码 |

## 3. 常见错误

1. 只盯代码目录，不读知识入口
2. 跳过 `00-meta` 直接开始联调
3. 只看 `changes/` 下 proposal，不看 `openspec/changes/` 下规格正文
4. 没有 test-cases 就直接实现
5. 修完 bug 不回写知识页
