# 新项目启动说明

## 这份文件给谁看

主要给 5 类角色看：

1. 项目负责人
2. 产品负责人
3. 架构 / 后端负责人
4. 前端 / 测试负责人
5. 参与协作的 AI Agent

## 启动目标

第一周不是把系统做完，而是跑通三件事：

1. 项目启动闭环
2. 第一条最薄业务切片
3. 第一套可复制的 change 模板与验证节奏

## 第一周建议节奏

### Day 1：冻结目标与边界

- 确认项目目标、范围、非范围
- 确认 RACI
- 补齐 `docs/00-meta/外部依赖与交接说明.md`
- 锁定权威 PRD、协议版本、架构来源

输出：

- Gate 0 `可立项`

### Day 2：梳理主流程与高风险规则

- 画清楚主流程、异常流程、回退流程
- 冻结租户、权限、历史、指令、幂等等高风险规则
- 补齐 `docs/01-overview/` 与 `docs/02-business-rules/`

输出：

- Gate 1 `可设计`

### Day 3：冻结架构、契约与外部依赖

- 明确遥测链路
- 明确指令链路
- 明确原始报文留存
- 明确模拟器与联调入口
- 明确回滚口径

输出：

- Gate 2 `可联调`

### Day 4：挑第一条最薄切片

建议第一条 change 同时满足：

- 有真实业务价值
- 有真实设备或模拟器
- 有真实页面或接口入口
- 有至少一条读链路和一条写链路
- 能演示历史、审计或回滚

### Day 5：冻结第一条 change 文档

至少补齐：

- `proposal.md`
- `spec.md`
- `design.md`
- `contracts/`
- `test-cases.md`
- `acceptance.md`
- `tasks.md`

输出：

- Gate 3 `可开发`

### Day 6：测试先行与实现

- 先写红态测试
- 再实现
- 再联调

### Day 7：验证、发布演练与回灌

- 补 `verification.md`
- 补 `rollout.md`
- 做一次发布与回滚演练
- 回灌测试场景和缺陷库

## 启动成功标准

至少满足：

1. 团队知道先读什么、先写什么、先确认什么。
2. 第一条真实链路能从入口走到数据落地与页面可见。
3. AI 与工程师使用同一套规则来源。
4. 外部依赖、环境、协议和版本锚点不悬空。
5. 第二个 change 可以沿用同一套模板继续做。
