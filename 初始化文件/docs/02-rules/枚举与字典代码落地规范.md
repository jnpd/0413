# 枚举与字典代码落地规范

## 1. 目的

把“固定业务选项不能硬编码”从原则落到代码层，统一后端枚举类、前端映射、字典接口和接口返回的实现方式。

## 2. 适用范围

适用于：

- 固定状态值
- 固定业务类型
- 固定操作类型
- 页面下拉选项
- 标签展示文案

不适用于：

- 用户可自由录入文本
- 临时调试开关
- 纯展示层、且会被运营频繁调整的可配置文案

## 3. 决策顺序

遇到一个“选项/状态/类型”时，按下面顺序判断：

1. 是否参与业务判断、状态流转、权限控制？
2. 值集合是否稳定？
3. 不同租户/项目是否允许配置差异？
4. 前后端是否都要使用同一套值？

结论：

- 稳定且参与业务判断：用枚举类/集中常量
- 主要用于展示、且允许配置差异：用字典表/字典接口
- 两者都需要：枚举为主，字典只做展示扩展，必须明确主来源

## 4. 后端落地要求

### 4.1 枚举类

固定业务枚举必须：

- 集中定义，不散落在 Controller / Service / Mapper XML
- 至少包含：
  - `code`
  - `label`
  - 必要时的 `description`
- 提供：
  - `fromCode(...)`
  - `isValid(...)`
  - `getLabelByCode(...)` 或等价能力

### 4.2 DTO / VO / Entity

- Entity 存储稳定枚举值，不存中文文案
- DTO 校验传入值是否合法
- VO 可返回：
  - 原始值
  - 展示文案
- 文案生成必须来自统一枚举或统一映射，不得手写三元表达式和魔法值

### 4.3 Mapper / SQL

禁止：

- 在 SQL 中直接硬编码中文文案
- 在 Mapper XML 中手写业务枚举转换逻辑

允许：

- 只查询原始 code
- 在应用层统一转 label

## 5. 前端落地要求

### 5.1 页面层

禁止在页面里直接出现：

- `status === 1 ? '启用' : '停用'`
- `type === 2 ? 'NB-IoT' : ...`
- 下拉 options 在多个页面各写一份

### 5.2 集中映射

前端必须至少有一层集中定义：

- `enum`
- `const map`
- `options factory`
- `dict adapter`

页面只消费集中定义，不直接重新解释 code。

### 5.3 查询区 / 表单 / 表格

- 查询区下拉、表单下拉、表格标签要共享同一来源
- 标签颜色、禁用状态、操作按钮可见性，应依赖统一枚举映射
- 详情页展示也必须复用同一映射

## 6. 字典接口使用要求

适合走字典接口的场景：

- 协议名称可能扩展
- 运营商名称可能调整
- 业务选项由后台配置管理

使用字典接口时：

- 前端保留 loading / error / fallback 处理
- 后端或前端要定义空值、缺失值兜底文案
- 若字典失败但业务仍要可用，需定义回退策略

## 7. 混合模式要求

如果一个字段同时有：

- 固定 code
- 可配置 label

则必须明确：

- 谁是主来源
- 谁负责合法性校验
- 谁负责展示补充

默认建议：

- code 合法性：枚举类
- label 展示：字典或映射表

## 8. 闭环检查清单

涉及枚举/字典改动时，至少检查：

- Java 枚举类或常量定义
- DTO 校验
- VO 返回字段
- Mapper / XML 是否仍有硬编码
- 前端 options 来源
- 表格列展示
- 详情展示
- 查询区下拉
- 表单下拉
- 标签颜色/禁用逻辑

## 9. 禁止事项

- 直接在 Controller / Service / 页面组件里写魔法值
- 同一含义多套 code / label 并存
- 后端返回一个 code，前端每个页面自己翻译一次
- 接口字段改了，枚举映射和展示不跟着改
