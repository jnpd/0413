# 02-rules 规则总览

## 1. 目录定位

`docs/02-rules/` 是当前仓库业务规则、项目级强约束、通用契约与实施说明的统一落盘目录。

- 当前仓库统一使用 `docs/02-rules/` 作为规则目录名
- 历史文档中的 `docs/02-business-rules/` 视为旧路径，不再继续使用
- 涉及字段、状态、权限、交互、接口、多租户等问题时，默认先进入本目录

## 2. 规则优先级

```text
feature contracts / openspec 真源
> docs/02-rules/ 中的项目级强规则
> docs/02-rules/ 中的通用契约
> docs/02-rules/ 中的实施说明 / 归档说明
> 外部原始资料或 starter 模板
```

补充说明：

- `前端实现规范.md`、`BladeX前端登录与仓库接口联调说明.md` 属于实施说明层，不得反向覆盖项目级强规则或通用契约
- 若某篇项目级强规则明确声明覆盖通用契约，以该项目级规则为准

### 3.3 通用契约

通用契约不是“没地方放的杂项规范”，而是跨两个及以上模块复用、用于裁决统一实现方式的公共口径。

分类原则不是按“页面名”分，而是按“它在裁决什么”分。

## 3.5 判定规则

### 3.5.1 什么算通用契约

同时满足以下条件时，优先归入通用契约：

- 至少会被两个及以上模块复用
- 裁决的是“统一实现方式”，不是某个模块自己的业务真相
- 不依赖某一张特定业务表、某一个固定流程或某一个 feature 私有状态机
- 可以被项目级强规则覆盖，但不会反过来覆盖项目级强规则

### 3.5.2 什么不算通用契约

出现以下任一情况时，不应归入通用契约：

- 文档在描述某个模块、某组字段、某条状态流转的真实业务真相
  这类应归入项目级强规则，或写入对应模块文档 / feature contracts
- 文档在描述历史联调口径、前端占位行为、外部原文归档
  这类应归入实施说明 / 归档说明
- 文档只服务于单个 change 或单个 feature
  这类应进入 `openspec/changes/<feature>/`

### 3.5.3 给 AI 的快速判断法

先问自己当前在裁判哪一类问题：

- “接口长什么样、文件怎么传、分页怎么回” -> 数据交换与集成类
- “页面怎么表现、提交中怎么拦、失败怎么提示” -> 页面体验与提交流程类
- “状态、权限、审计怎么不打架” -> 业务表达与控制类
- “项目启动前哪些技术前提必须先定死” -> 工程基线类
- “这个模块自己的真实业务规则是什么” -> 不属于通用契约，转项目级强规则或模块文档

## 4. 推荐阅读顺序

### 4.1 人工阅读

1. 先看 feature 的 `openspec/changes/<feature>/contracts/` 或对应 canonical spec
2. 再看本页确认文档层级
3. 再进 [业务规则选路索引.md](./业务规则选路索引.md)
4. 先判断命中的文档属于哪一类，再阅读对应项目级强规则与通用契约
5. 只有需要落地实现、联调或回放历史口径时，再读实施说明层

### 4.2 AI 执行

- 命中业务规则问题时，先回报“本次命中了哪些规则页”
- 若项目级强规则与通用契约冲突，优先按项目级强规则执行
- 若实施说明与上层规则冲突，优先按上层规则执行，并在回复中显式说明

## 5. 维护要求

- 本目录新增文档时，必须补机器可读头信息：`类型`、`触发场景`、`联读文档`、`是否覆盖通用契约`
- 新增规则前先确认它属于哪一层，避免把实施说明写成项目级强规则
- 任何路径引用统一使用 `docs/02-rules/`，不要再写 `docs/02-business-rules/`
