# 交付护栏契约

## 1. 目的

统一开发过程中的硬约束，减少 AI 扩散范围和返工。

## 2. 固定护栏

- 一次 change 只能有一个主目标
- 本次做什么必须明确
- 本次不做什么必须明确
- 页面开工前必须先有契约
- 接口返回结构必须先明确
- 表单字段、校验、交互结果必须先明确
- 固定状态值、固定业务选项必须先明确由枚举类还是字典承担
- 权限、异常、空态必须先明确归属
- 先主链，再边界
- 先最小可用，再补完整体验
- 先复用公共能力，再补局部实现
- 基线不过，不算完成
- smoke 没跑，不算完成
- 没有验证证据，不宣称完成

## 3. 闭环修改要求

只要涉及业务字段、状态、枚举、流程的新增 / 修改 / 删除，默认必须先做 impact analysis，再实施修改。

### 3.1 默认检查范围

- 数据层：MySQL、Entity、DTO、VO、Mapper、Mapper XML
- 后端链路：Controller、Service、保存、更新、详情、查询、导出
- 前端链路：新增、编辑、详情、查询区、列表列、字典映射、校验
- 一致性：默认值、空值、历史兼容、展示文案、枚举来源

### 3.2 默认全局搜索项

- 字段名
- 中文标题 / label
- 接口路径
- DTO / VO / Entity / Mapper XML 关联定义
- 前端列定义、表单定义、查询定义或等价配置
- 枚举类、字典映射、展示文案来源

### 3.3 禁止事项

- 只改新增，不查编辑和详情
- 只改接口，不查列表返回字段
- 只改后端，不查前端表单和枚举展示
- 只改字段值，不查默认值、空值与历史兼容
- 直接在页面、SQL、接口转换层手写魔法值和中文枚举文案

### 3.4 完成标准

回复或交付说明中必须包含：

1. impact analysis 涉及点
2. 已修改文件和链路
3. 已检查但无需修改的关联点
4. 剩余风险或待确认项

## 4. 适用方式

- 作为新项目启动前的固定规则
- 作为每个 change 开工前的检查单
- 作为 AI 接收任务时的默认约束

## 5. AI 使用要求

- 没有项目级覆盖时，默认直接执行这组护栏
- 如果用户要突破护栏，必须在 change 文档里显式说明原因
