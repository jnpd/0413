# 系统管理模块 - 数据库设计

> 本文档按当前实体、Mapper 和升级 SQL 校准，描述系统管理模块真实依赖的数据表，而不是历史方案里的抽象分域设计。

## 1. 代码依据

主要代码落点：

- `modules/system/pojo/entity/*`
- `modules/system/enterprise/pojo/entity/*`
- `modules/desk/pojo/entity/*`
- `modules/desk/mapper/NoticeMapper.xml`
- `doc/sql/upgrade/water-dev.mysql.enterprise-management.sql`

## 2. 核心表

| 表名 | 作用 | 当前页面/能力 |

| `biz_enterprise` | 企业主表 | 企业管理 |
| `biz_user_enterprise` | 用户-企业关系与管理员标识 | 人员管理、企业权限范围 |

| `blade_role_scope` | 角色数据权限范围 | 后端支持，前端未接 |
| `blade_scope_data` | 数据权限定义 | 后端支持，前端未接 |
| `blade_scope_api` | API 权限定义 | 后端支持，前端未接 |

## 3. 关键字段

### 3.1 企业相关

`biz_enterprise` 当前关键字段：

- `enterprise_id`
- `tenant_id`
- `enterprise_name`
- `enterprise_code`
- `enterprise_type`
- `admin_user_id`
- `status`
- `is_deleted`

`biz_user_enterprise` 当前关键字段：

- `user_id`
- `enterprise_id`
- `is_admin`
- `status`
- `is_deleted`

说明：

- 企业编码不是页面录入字段，后端按 `QY + enterpriseId` 生成
- 企业管理员关系由 `admin_user_id` 和 `biz_user_enterprise.is_admin` 共同描述

### 3.2 用户与角色相关

`blade_user` 当前与本模块直接相关的字段：

- `id`
- `tenant_id`
- `account`
- `real_name`
- `role_id`
- `enterprise_id`
- `status`

`blade_role` 当前与本模块直接相关的字段：

- `id`
- `tenant_id`
- `role_name`
- `role_alias`
- `remark`
- `enterprise_id`
- `role_source`

说明：

- `enterprise_id` 决定角色是否为企业定制角色
- `role_source` 参与角色可见性与角色下拉过滤

### 3.3 菜单与权限相关

`blade_menu` 当前主要承担三类数据：

- 左侧菜单
- 动态路由
- 按钮权限

`blade_role_menu` 用于角色授权结果落表。

当前前端只消费：

- 菜单树 / 路由
- 按钮权限

前端没有独立菜单维护页面。

### 3.4 公告相关

`blade_notice` 当前关键字段：

- `id`
- `title`
- `category`
- `release_time`
- `content`
- `status`
- `is_deleted`

`biz_notice_profile` 当前关键字段：

- `notice_id`
- `priority`
- `summary`
- `scope_type`
- `scope_tenant_type`
- `expire_time`
- `status`
- `is_deleted`

说明：

- `summary` 由后端从富文本正文自动提取
- 当前页面不支持人工维护 `priority`、`scope_type`、`expire_time`
- `biz_notice_read` 已有升级 SQL，但当前功能未读写

### 3.5 表格列配置相关

`blade_user_table_column_setting` 当前关键字段：

- `id`
- `tenant_id`
- `user_id`
- `pathname`
- `table_type`
- `ordered_ids`
- `create_time`
- `update_time`

说明：

- 唯一键为 `tenant_id + user_id + pathname + table_type`
- `ordered_ids` 直接存当前显示列 key 的有序 JSON 数组
- 查不到记录时后端返回空数组 JSON，前端回退默认列顺序

## 4. 关系说明

### 4.1 企业与用户

- `biz_enterprise.admin_user_id -> blade_user.id`
- `biz_user_enterprise.user_id -> blade_user.id`
- `biz_user_enterprise.enterprise_id -> biz_enterprise.enterprise_id`

### 4.2 用户与角色

- `blade_user.role_id -> blade_role.id`
- 人员页的“角色配置”最终调用 `/blade-system/user/grant`

### 4.3 角色与菜单

- `blade_role.id -> blade_role_menu.role_id`
- `blade_menu.id -> blade_role_menu.menu_id`

### 4.4 公告

- `blade_notice.id -> biz_notice_profile.notice_id`
- `blade_notice.id -> biz_notice_read.notice_id`（当前未使用）

### 4.5 表格列配置

- `blade_user_table_column_setting.tenant_id` 用于租户隔离
- `blade_user_table_column_setting.user_id -> blade_user.id`
- 同页多表依赖 `pathname + table_type` 做细分

## 5. 当前代码约束

### 5.1 企业管理

- 新增企业时，后端同时创建企业管理员账号
- 删除企业时，若仍有有效的非管理员人员账号，后端阻止删除
- 企业数据按 `tenant_id` 与 `enterprise_id` 做访问隔离

### 5.2 人员管理

- 非平台超管只能操作自己所属企业的数据
- 企业管理员角色会按企业类型自动解析，不由页面自由输入
- 主管理员记录不会在前端暴露编辑/删除入口

### 5.3 角色管理

- 平台超管可见全部角色
- 企业侧只可管理企业自定义角色
- 前端只保存菜单权限，不保存数据权限与 API 权限

### 5.4 公告管理

- 新增/编辑公告时，标题和正文必填
- 正文会做 HTML 清洗，自动生成摘要
- 公告类型当前固定为 `notice`
- 公告优先级当前默认普通优先级

### 5.5 表格列配置

- 前端不得上传用户 ID，后端必须从安全上下文取当前登录人
- 保存接口按唯一键执行 upsert 语义
- 请求头必须带齐租户、部门、角色上下文，避免系统侧接口取不到当前会话信息

## 6. 当前与旧文档的差异修正

本模块当前真实表范围并不是旧文档里泛写的“BladeX 内置表集合 + 若干扩展表全量覆盖”，而是：

- 前端页面实际管理：企业、人员、角色、公告
- 动态菜单与顶部菜单真实依赖：`blade_menu`、`blade_top_menu`
- 公告扩展真实依赖：`biz_notice_profile`
- 公告已读表：已建表但未使用

因此本文件只保留与当前代码直接相关的表和约束。
