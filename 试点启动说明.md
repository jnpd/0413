# WaterManger 知识底座试点启动说明

## 1. 这份文件给谁看

这份说明给 4 类人看：

1. 试点负责人
2. 产品 / 后端 / 前端 / 测试 owner
3. 接手 WaterManger 的新成员
4. 参与需求分析、设计、测试和实现的 Agent

目标只有一个：

> 拿到这个知识底座目录后，不需要自己猜顺序，就知道第一周怎么启动。

## 2. 试点目标

本次不是一次性重写所有文档，而是先完成两件事：

1. 把 WaterManger 现有知识整理成 starter 风格入口
2. 让每个已存在的 change 具备可继续执行的 proposal 镜像入口和 `openspec` 规格包

首个推荐样板 feature：

- `testing-history-inspection-closure`

原因：

- 规则最密集
- 牵涉历史只读、绑定唯一、删除守卫、企业隔离
- 最适合检验“知识先于代码”的流程

## 4. 第一周做什么

### Day 1

- 确认负责人和分工
- 补齐交接入口中的代码仓、环境、PRD 锚点
- 确认本次试点只先按知识底座和规格流转推进

### Day 2

- 阅读项目概览、模块地图、系统架构、数据模型
- 确认高风险规则页

### Day 3

- 逐个检查 `changes/2026-04/*/proposal.md`
- 选定当前 active feature

### Day 4

- 阅读对应 `openspec/changes/<feature>/`
- 确认 contracts 和 test-cases

### Day 5

只有这些条件满足，才进入实现：

1. proposal 已确认
2. spec 已确认
3. design 已确认
4. contracts 已确认
5. test-cases 已确认
6. acceptance 已确认

## 5. 这次试点不能放松的点

1. 历史只读
2. 绑定关系唯一
3. 删除守卫生效
4. 企业隔离由后端强制执行
5. 包装与打印事件不能污染主状态
