# Enterprise Management Logic Alignment Design

## 1. 背景

当前 [`react-auth-rbac-foundation`](/E:/demo0413/openspec/changes/react-auth-rbac-foundation/specs/system-management-foundation/spec.md) 把企业管理描述成“当前仓库无企业管理扩展接口，因此只展示占位页”。这与系统管理模块文档中已经冻结的业务事实冲突：

- 企业管理是实际存在的“企业 + 企业管理员”一体化页面
- 页面具备查询、筛选、分页、新增、编辑、删除能力
- 保存与删除都带有明确的后端业务约束
- 企业数据必须受权限边界与租户隔离规则约束

因此，本 change 的目标不是发明新流程，而是把已经存在但被错误描述的企业管理逻辑重新冻结进 canonical spec。

## 2. 设计目标

- 让 enterprise 模块从“占位页口径”回到“真实业务页口径”
- 保持现有企业管理界面不变，不靠改布局解决逻辑偏差
- 冻结新增、编辑、删除、权限、租户隔离、企业管理员联动语义
- 明确与人员管理、角色管理的联动边界，避免只改单点

## 3. 非目标

- 不重做企业管理 UI
- 不改企业管理入口名称、tab 结构、列表列顺序、表单字段顺序
- 不新增长链路审批、企业详情页、批量操作
- 不扩展未被文档确认的企业字段

## 4. 设计方案

### 4.1 页面边界与 UI 稳定性

企业管理页继续沿用当前页面形态，不拆新页，不改布局，不改用户已习惯的可见字段集合。

本 change 对“界面保持不变”的解释是：

- 不新增新的视觉模块、筛选区块、步骤条、详情抽屉
- 不调整列表列顺序
- 不调整表单字段顺序
- 允许只修正现有控件的行为语义

如果当前实现中存在与模块文档冲突的交互语义，例如企业编码被当作可编辑输入框，本次也只能通过“保留位置但改成只读/回显”来修正，不能通过删除该字段或重新设计弹窗来解决。

### 4.2 列表查询语义

企业管理页的查询能力冻结为：

- 关键字搜索：同时覆盖企业名称、管理员姓名、登录账号
- 企业类型筛选
- 状态筛选
- 分页查询

企业列表展示语义冻结为：

- 企业名称
- 企业类型
- 管理员姓名
- 登录账号
- 初始密码掩码
- 创建时间
- 状态

如实现中额外回显企业编码，可保留，但不能要求用户在保存时手工维护该值。

### 4.3 保存语义

新增 / 编辑企业共用同一保存入口，但语义不同。

新增企业时：

- 必须同时保存企业主记录与企业管理员账号
- 企业编码由后端按 `QY + enterpriseId` 生成
- 企业管理员关系必须同时体现在：
  - `biz_enterprise.admin_user_id`
  - `biz_user_enterprise.is_admin`
- 新增失败时必须整体回滚，不能出现“企业建成但管理员没建好”或“管理员建成但企业关系缺失”的半成功状态

编辑企业时：

- 允许修改企业名称、企业类型、管理员姓名、登录账号
- 密码字段留空表示不修改
- 只有在明确输入新密码时，才触发管理员密码重置
- 已有关联关系必须保持一致，不能只改企业表或只改用户企业关系表

### 4.4 企业编码语义

企业编码不属于前端录入字段。

冻结规则：

- 创建时由后端生成
- 编辑时仅允许回显，不允许手工修改
- 前端不得把企业编码作为必填输入项提交
- 若界面中保留企业编码字段，该字段只能承载“展示已有值”这一语义

这条规则用于在不改界面的前提下，纠正“企业编码被当作人工输入字段”的逻辑偏差。

### 4.5 删除守卫

删除企业属于高风险动作，必须同时满足权限、租户和业务守卫三层约束。

冻结规则：

- 新增 / 删除按钮仅平台超管可见
- 删除前必须校验目标企业仍在当前可操作租户范围内
- 若该企业下仍存在有效的非管理员人员账号，则删除必须被阻止
- 删除失败时必须返回稳定、可读的业务提示，不能伪装成“删除成功但列表没刷新”

成功删除后，企业主表、用户企业关系、管理员归属不能留下孤儿引用。管理员账号的最终清理策略沿用后端实现口径，但结果上不能留下仍指向已删除企业的有效关联。

### 4.6 权限与租户隔离

企业管理同时命中 [`权限边界规则.md`](/E:/demo0413/docs/02-rules/权限边界规则.md) 与 [`多租户隔离规则.md`](/E:/demo0413/docs/02-rules/多租户隔离规则.md)。

冻结规则：

- 按钮权限码：
  - `enterprise_add` / `tenant_add`
  - `enterprise_edit` / `tenant_edit`
  - `enterprise_delete` / `tenant_delete`
- 前端按钮显示只做体验优化，不承担最终安全责任
- 后端必须基于安全上下文裁定租户范围，不能以前端传入的租户参数为准
- 查询、保存、删除都必须做租户归属校验
- 越权时返回明确拒绝语义，不能伪装为“数据不存在”

### 4.7 与人员 / 角色模块的联动

企业管理不是孤立模块，它与人员管理、角色管理存在强耦合关系。

冻结联动：

- 企业管理员是人员管理中的一种特殊人员
- 企业管理员角色不能自由选择，而是按企业类型自动解析
- 删除守卫依赖“该企业下是否仍存在有效的非管理员人员账号”
- 企业类型变化若影响管理员角色映射，必须在保存链路中同步处理，不能让企业页与人员页口径分裂

这意味着实现阶段不能只修改企业表单提交，还必须检查：

- `biz_enterprise`
- `biz_user_enterprise`
- `blade_user`
- 角色解析逻辑
- 人员页列表与详情回填

### 4.8 交互与错误反馈

本 change 不改界面，但要求界面行为遵守项目级交互契约。

冻结规则：

- 所有新增、编辑、删除写操作必须显示整体界面阻断式 loading
- 提交失败时保留用户输入
- 成功后关闭弹窗并刷新列表
- 错误提示优先透传后端业务文案
- 删除守卫、越权、参数错误必须是用户可读提示，而不是通用“操作失败”

## 5. 实施影响面

该 change 的实现默认要做闭环检查，至少覆盖：

- 数据层：
  - SQL / 表结构
  - Entity
  - DTO
  - VO
  - Mapper
  - Mapper XML
- 后端链路：
  - Controller
  - Service
  - 新增 / 编辑 / 删除 / 详情 / 查询
- 前端链路：
  - 查询区
  - 列表列
  - 新增弹窗
  - 编辑回填
  - 错误提示
  - loading
- 一致性：
  - 企业编码生成
  - 管理员关系同步
  - 状态回显
  - 历史数据兼容

## 6. 风险与待确认

- 当前仓库搜索未直接定位到企业管理后端代码实现，因此设计以模块文档为真源，代码落地时要做一次事实校准。
- 企业类型的底层枚举值与管理员角色映射编码，当前文档只给出业务语义，未冻结完整代码表；实现阶段需按实际字典 / 角色数据再次对齐。
- 删除成功后的管理员账号处置方式，当前文档未给出完整细节；本 change 仅冻结“不能留下有效孤儿关系”这一结果要求。
