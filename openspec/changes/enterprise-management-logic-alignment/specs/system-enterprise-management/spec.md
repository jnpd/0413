# 系统管理企业模块规格

## 状态

草稿，已可进入实现规划。

## 需求

### 需求 1. 企业管理必须被定义为真实业务模块

系统管理中的企业管理页必须被定义为真实的企业管理模块，不能继续被当作“后端缺口占位页”。

#### 场景：操作人打开企业管理页

- 前提：操作人具备系统管理页面访问权限
- 当：操作人打开企业管理页
- 则：页面应被视为真实的企业列表与表单工作流
- 且：系统不得把企业管理渲染成假占位页

### 需求 2. 界面保持不变，只修正逻辑

本次企业管理变更必须在保持现有页面结构和可见交互布局不变的前提下，修正业务逻辑和契约行为。

#### 场景：企业管理逻辑被校准

- 前提：企业管理页已经存在既定的列表和弹窗布局
- 当：企业管理逻辑被修正
- 则：页面仍保持相同的可见布局、字段顺序和列表结构
- 且：只修正业务行为、校验、权限与数据绑定逻辑

### 需求 3. 企业保存必须管理“企业 + 企业管理员”一体化关系

企业保存流程必须把企业数据和企业管理员数据视为同一条业务事务。

#### 场景：新增企业

- 前提：操作人提交一条新企业数据
- 当：保存成功
- 则：后端创建企业主记录
- 且：创建企业管理员账号
- 且：创建用户与企业关系
- 且：保持 `admin_user_id` 与 `is_admin` 语义一致

#### 场景：编辑企业且不重置密码

- 前提：操作人编辑一条已存在的企业数据
- 且：密码字段为空
- 当：保存成功
- 则：企业字段被更新
- 且：管理员密码不被重置

#### 场景：编辑企业并重置密码

- 前提：操作人编辑一条已存在的企业数据
- 且：输入了新密码
- 当：保存成功
- 则：企业字段被更新
- 且：管理员密码被重置

### 需求 4. 企业编码必须由后端生成

企业编码必须由后端生成，不能被当作用户手工维护的输入字段。

#### 场景：新增企业后生成企业编码

- 前提：操作人新增企业
- 当：企业保存成功
- 则：后端根据企业标识生成企业编码

#### 场景：界面中展示企业编码

- 前提：页面仍保留企业编码字段或列表列
- 当：操作人查看或编辑企业数据
- 则：该字段只承担展示值或只读回显语义
- 且：前端不要求用户手工录入企业编码

### 需求 5. 删除必须遵守企业人员守卫

当企业下仍存在有效的非管理员人员账号时，企业删除必须被阻止。

#### 场景：存在有效非管理员人员时删除被阻止

- 前提：目标企业下仍存在有效的非管理员人员账号
- 当：操作人尝试删除该企业
- 则：删除请求被拒绝
- 且：操作人收到可读的业务错误提示

#### 场景：清除守卫后允许删除

- 前提：目标企业下已不存在有效的非管理员人员账号
- 当：操作人删除该企业
- 则：企业按后端既定策略被删除
- 且：不得留下仍然有效的孤儿用户企业关系

### 需求 6. 企业权限必须同时遵守按钮码和平台角色约束

企业管理动作必须同时遵守按钮级权限码与平台超管约束。

#### 场景：平台超管进入企业管理页

- 前提：当前用户是平台超管
- 当：企业管理页渲染
- 则：新增和删除动作可用，但仍受按钮权限码控制

#### 场景：非平台用户进入企业管理页

- 前提：当前用户不是平台超管
- 当：企业管理页渲染
- 则：新增和删除动作不对其暴露

#### 场景：操作人缺少编辑权限

- 前提：当前权限集中不包含 `enterprise_edit` 或 `tenant_edit`
- 当：企业管理页渲染
- 则：编辑动作保持隐藏或禁用
- 且：即使强行调用后端写接口，也仍会被后端拒绝

### 需求 7. 企业查询和写入必须遵守租户隔离

企业管理的查询和写入都必须以后端租户上下文为准。

#### 场景：在租户范围内查询

- 前提：当前用户属于某一租户
- 当：用户查询企业数据
- 则：返回结果只包含其授权租户范围内的企业数据

#### 场景：跨租户写入

- 前提：当前用户尝试编辑或删除授权租户范围外的企业数据
- 当：请求到达后端
- 则：后端以稳定的越权错误语义拒绝该请求

### 需求 8. 企业写操作必须遵守项目级提交契约

企业新增、编辑、删除必须遵守项目级表单提交与加载态契约。

#### 场景：保存进行中

- 前提：操作人提交新增或编辑
- 当：请求仍在处理中
- 则：页面或弹窗显示阻断式写操作加载态
- 且：禁止重复提交与关闭动作

#### 场景：保存失败

- 前提：企业保存因业务错误或校验错误而失败
- 当：页面处理失败响应
- 则：当前用户输入被保留
- 且：页面优先展示后端返回的可读错误文案
