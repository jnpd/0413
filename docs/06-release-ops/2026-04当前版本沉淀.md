# 2026-04 当前版本沉淀

## 文档信息

| 项 | 内容 |
|---|---|
| 目标 | 给后续项目复用当前版本已经验证过的方法、结构和资产 |
| 适用范围 | 知识底座建设、React 后台原型接 BladeX 鉴权/RBAC、系统管理闭环治理 |
| 最后更新 | 2026-04-15 |

## 1. 一句话结论

当前版本不是“零散修了几个页面”，而是完成了三件底座级改造：

- 把项目从“文档、代码、规则各说各话”收敛成 `README -> REPO_WIKI -> docs -> changes/openspec` 的真源体系
- 把前端从“假登录 + 本地切页原型”升级成“真实登录 + 刷新 token + 动态菜单/路由 + 按钮权限”的可联调后台壳
- 把系统管理从“页面演示”拉回“企业、人员、角色、菜单权限、租户边界、删除守卫”的真实业务闭环

## 2. 从第一性原理看，这版到底在解决什么

### 2.1 原始问题

后台项目从原型走向可联调，通常会同时踩三类坑：

| 原始问题 | 表面症状 | 本质 |
|---|---|---|
| 真源混乱 | PRD、规则页、代码、AI spec 口径打架 | 团队没有先回答“什么才算事实” |
| 鉴权半接入 | 登录通了，但刷新、菜单、按钮权限、恢复会话断裂 | 鉴权被当成单接口接入，而不是完整生命周期 |
| 系统管理被页面化理解 | 企业、人员、角色被拆成三个孤立 Tab | 实际上这是同一套身份、组织、权限模型 |

### 2.2 当前版本的裁决

这版的核心决策，不是继续补零碎功能，而是先修正三个根问题：

1. 先修“真源”，再修实现。没有真源顺序，越写 spec 越容易跑偏。
2. 把鉴权当闭环，不当页面。登录、存 token、刷新、恢复、菜单、按钮权限必须一次打通。
3. 把系统管理当模型，不当页面。企业、管理员、普通人员、角色、菜单授权是同一张关系网。

## 3. 当前版本实际改了什么

### 3.1 知识与规格层

当前版本把仓库从“资料散落”重组为可以交接和复用的知识底座：

- 明确了仓库入口和阅读顺序：`README.md`、`REPO_WIKI.md`、`docs/00-07/`
- 把变更拆成时间入口和 canonical spec 真源：
  - `changes/2026-04/react-auth-rbac-foundation/proposal.md`
  - `changes/2026-04/enterprise-management-logic-alignment/proposal.md`
  - `openspec/changes/<feature>/...`
- 增补了长期护栏：
  - `docs/00-meta/change-writing-anti-drift-guide.md`
  - `docs/07-incidents/2026-04-15-system-management-change-drift.md`
  - `scripts/check_change_guardrails.py`

这一层的沉淀价值不是“文档变多了”，而是形成了可执行的写法约束：

- proposal 负责范围声明
- openspec 负责 canonical spec
- foundation 类 change 必须声明“只冻结基础能力，不替代完整业务真相”
- 发现“没搜到代码”时，只能写“待校准”，不能直接写“能力不存在”

### 3.2 前端鉴权与导航层

前端从原来的本地状态原型，升级成了可联调的 React 后台壳：

- 真实登录链路：
  - `POST /blade-auth/oauth/token`
  - `GET /blade-auth/oauth/user-info`
  - `GET /blade-auth/oauth/logout`
- 统一请求层：
  - `frontend/src/lib/http/bladeClient.ts`
  - 自动补 `Authorization`、`Blade-Auth`、`Blade-Requested-With`
  - 透传 `Tenant-Id`、`Dept-Id`、`Role-Id`
  - 401 单飞刷新与失败请求重放
- 会话层：
  - `frontend/src/auth/authApi.ts`
  - `frontend/src/auth/sessionAdapter.ts`
  - `frontend/src/auth/sessionStorage.ts`
- 动态导航层：
  - `frontend/src/router/navigationApi.ts`
  - `frontend/src/router/navigationBootstrap.ts`
  - `frontend/src/router/menuAdapter.ts`
  - `frontend/src/router/routeRegistry.tsx`
- 页面壳改造：
  - `frontend/src/App.tsx`
  - 从 `isLoggedIn/currentView` 切页改为真实路由和恢复会话启动流

这部分可复用的抽象是：

- 先建 HTTP/auth/router/permission 基础层，再接具体业务页
- 菜单树必须经 adapter 归一化后再喂给 UI，不能直接把后端结构硬塞进侧边栏
- token 刷新必须做单飞，否则一旦 401 并发就会把会话系统打崩

### 3.3 系统管理业务闭环层

当前版本没有停留在“把页面接上接口”，而是把系统管理的核心关系重新拉直：

- 企业管理后端闭环新增：
  - `bladex-boot/.../system/controller/EnterpriseController.java`
  - `bladex-boot/.../system/service/impl/EnterpriseServiceImpl.java`
  - `bladex-boot/.../system/mapper/EnterpriseMapper.java`
  - `bladex-boot/.../system/mapper/EnterpriseMapper.xml`
  - `bladex-boot/.../system/pojo/dto/EnterpriseSubmitDTO.java`
  - `bladex-boot/.../system/pojo/entity/Enterprise.java`
  - `bladex-boot/.../system/pojo/entity/UserEnterprise.java`
  - `bladex-boot/.../system/pojo/vo/EnterpriseVO.java`
- 前端企业页从占位改成真实 CRUD：
  - `frontend/src/features/system/enterprise/EnterpriseManagementTab.tsx`
- 人员、角色同步校准：
  - `frontend/src/features/system/user/UserManagementTab.tsx`
  - `frontend/src/features/system/role/RoleManagementTab.tsx`
  - `bladex-boot/.../system/controller/UserController.java`
  - `bladex-boot/.../system/controller/RoleController.java`

这一层冻结下来的真实规则包括：

- 企业新增时同步创建企业管理员
- 企业编码后端生成，不允许前端手工编辑
- 编辑企业时密码留空表示不重置
- 删除企业前要检查是否还有有效非管理员人员账号
- 新增/删除企业仅平台超管可见
- 非平台管理员在角色分页里排除 `administrator`、`admin`、`water_company_admin`、`water_factory_admin`
- 角色授权当前已形成“读取授权树 -> 勾选菜单 -> 保存授权”的菜单权限闭环

对别的项目最有价值的不是字段名，而是这个闭环模型：

- 组织实体
- 组织管理员
- 普通成员
- 角色集合
- 菜单权限
- 租户边界
- 删除守卫

只要是企业后台、SaaS 后台、代理商后台，这个模型都能迁移。

### 3.4 验证与防回归层

当前版本不是只靠人工口述验收，而是补了前后端验证资产：

- 前端测试：
  - `frontend/src/test/auth-workflow.test.ts`
  - `frontend/src/test/navigation-bootstrap.test.ts`
  - `frontend/src/test/system-management-contracts.test.ts`
- 后端契约测试：
  - `bladex-boot/.../EnterpriseControllerContractTest.java`
  - `bladex-boot/.../RoleControllerPageMappingTest.java`

这说明这版已经把“能力”转成了“可回归的契约”：

- 登录头、token 刷新、用户上下文映射可测
- 菜单/按钮权限归一化可测
- 企业、用户、角色分页和授权接口锚点可测

## 4. 这版真正产出的可复用资产

如果给别的项目复用，不建议照搬整个仓库，建议抽下面四组资产：

### 4.1 知识治理资产

- `changes/<YYYY-MM>/<feature>/proposal.md` + `openspec/changes/<feature>/` 的双层结构
- `change-writing-anti-drift-guide.md`
- `scripts/check_change_guardrails.py`

适用场景：

- 多人协作、AI 协作下的需求/spec 容易漂移项目

### 4.2 React 鉴权/RBAC 底座

- `frontend/src/lib/http/bladeClient.ts`
- `frontend/src/auth/*`
- `frontend/src/router/*`
- `frontend/src/permissions/*`

适用场景：

- 已有 Blade/Saber/若依类后端，前端是 React/Vite 原型壳

### 4.3 系统管理闭环模型

- 企业、人员、角色、菜单授权的联动关系
- 平台超管与租户内角色的边界拆分
- 删除守卫和管理员角色自动解析

适用场景：

- 企业后台、租户后台、组织权限后台

### 4.4 迁移作业手册

- `skills/migrating-saber-auth-rbac-to-react/SKILL.md`

它不是单纯提示词，而是把“从旧后台迁移登录/菜单/权限到底该按什么顺序做”固化成了流程：

1. 先盘点 source/target
2. 先补 substrate，再接页面
3. 菜单必须先 adapter
4. auth 必须做全生命周期
5. 权限要同时覆盖路由和按钮

## 5. 当前版本的边界

这版很重要，但并不是“整个后台都完成了”，边界也要说清楚：

- Ant Design 后台模板尚未正式落地
- 角色管理前端当前只暴露菜单权限树，数据权限树与 API 权限树还未完全上屏
- 业务大页里仍有原型/静态部分，当前重点是先把系统管理底座打通
- 环境脱敏说明、RACI、完整发布平台说明仍待补齐

## 6. 给别的项目的直接迁移建议

如果别的项目要复用这版成果，推荐按下面顺序，而不是一次性照搬：

1. 先复制知识治理骨架：`README/REPO_WIKI/docs/changes/openspec`
2. 再复制前端 `http + auth + router + permission` 四层底座
3. 然后选一个高价值管理域做闭环样板，优先是“组织/用户/角色”
4. 最后再扩散到仓库、批次、测试、包装等业务页

根本原因很简单：

- 没有真源，后续会继续写漂
- 没有 auth/rbac 底座，业务页接得越多，返工越大
- 没有闭环样板，团队会继续按“单页接接口”的思路重复犯错
