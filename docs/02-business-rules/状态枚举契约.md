# 状态枚举契约

## 1. 目的

统一状态值、状态展示、状态流转的表达方式，避免前后端各写一套。

## 2. 固定定义方式

每个状态枚举至少明确：

- `value`: 真正存储值
- `label`: 页面展示文案
- `tagType`: 标签视觉类型，可选
- `allowedActions`: 当前状态允许的动作
- `nextStates`: 当前状态允许流转到的状态

## 3. 固定约束

- 状态值只能有一套 source of truth
- 前端只负责展示和禁用控制，不负责定义业务流转规则
- 后端必须校验状态流转是否合法
- 新状态加入时，必须同步更新文档、常量和测试
- 禁止在 Controller、Service、Mapper XML、前端页面、表格列、表单校验中直接硬编码枚举值或中文展示文案
- 固定业务枚举必须集中定义在统一枚举类或统一映射常量中，不允许同一含义出现多套魔法值

## 4. 枚举类与字典分工

### 4.1 固定枚举

满足以下条件时，优先使用枚举类或集中常量：

- 值集合稳定
- 由代码和业务规则共同约束
- 需要参与状态流转、权限判断、按钮可用性判断

示例：

- 库存状态
- 测试状态
- 任务状态
- 命令执行状态

### 4.2 可配置枚举

满足以下条件时，优先使用字典表或字典接口：

- 文案或可选项可能由运营配置
- 不同租户或不同项目存在可配置差异
- 仅影响展示或可选项，不承载核心状态机

### 4.3 禁止混用

- 同一业务含义不能一部分写死在枚举类，一部分又从字典表读取
- 如果既有枚举类又有字典展示，必须明确谁是主来源、谁是展示映射

## 5. 标准示例

```json
[
  {
    "value": "draft",
    "label": "草稿",
    "tagType": "default",
    "allowedActions": ["edit", "submit", "delete"],
    "nextStates": ["submitted"]
  },
  {
    "value": "submitted",
    "label": "已提交",
    "tagType": "processing",
    "allowedActions": ["view", "revoke"],
    "nextStates": ["draft", "approved", "rejected"]
  }
]
```

## 6. AI 使用要求

- 只要出现状态字段，就默认要求补状态枚举表
- 没有状态流转表时，不允许 AI 自己猜哪些按钮可点、哪些状态可跳
- 只要出现固定状态值或固定业务选项，就默认检查是否应抽成枚举类，而不是在代码中硬编码
