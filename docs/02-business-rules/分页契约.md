# 分页契约

## 1. 目的

统一列表查询的分页入参与分页返回，避免每个页面、每个模块重新约定。

## 2. 固定入参

- `pageNum`: 页码，从 `1` 开始
- `pageSize`: 每页条数
- `keyword`: 通用关键字搜索，可选
- `sortField`: 排序字段，可选
- `sortOrder`: 排序方向，可选，取值 `asc` 或 `desc`

## 3. 固定出参

- `pageNum`: 当前页码
- `pageSize`: 当前每页条数
- `total`: 总条数
- `records`: 当前页记录列表

## 4. 固定约束

- 不允许同一项目同时出现 `current/pageNo/pageNum` 多套字段
- `records` 为空时返回空数组，不返回 `null`
- `total` 必须返回数字
- 列表页刷新、筛选、分页时都复用这套结构

## 5. 标准示例

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "pageNum": 1,
    "pageSize": 20,
    "total": 128,
    "records": [
      {
        "id": 1001,
        "name": "A-001"
      }
    ]
  },
  "traceId": "7f1f7b83b7e94c31"
}
```

## 6. AI 使用要求

- 只要出现后台列表页，默认优先套用这份契约
- 如果项目需要游标分页或流式分页，必须在项目级文档里明确覆盖
