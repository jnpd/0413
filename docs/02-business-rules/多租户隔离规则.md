# 多租户隔离规则

## 文档信息

| 项 | 内容 |
|---|---|
| 最后更新 | 2026-04-13 |
| 风险级别 | 高 |

## 1. 目标

冻结本项目的租户隔离口径，避免前后端、SQL 和测试各自理解一套。

## 2. 当前已验证基础

### 2.1 字段口径

- `blade.tenant.column = tenant_id`
- `tb_device.tenant_id` 已存在
- `tb_meter_latest.tenant_id` 已存在

### 2.2 框架口径

- BladeX 多租户增强已开启
- `blade_user` 当前在 `exclude-tables` 中

## 3. 项目级隔离原则

### 3.1 查询隔离

- 业务查询默认以后端租户上下文为准。
- 前端传入的租户参数不应成为最终过滤依据。
- 任何跨租户查询都必须有明确的平台级豁免说明。

### 3.2 写入隔离

- 新增、编辑、删除、出库、导入等写操作必须在后端校验目标数据归属。
- 不能依赖前端是否隐藏按钮来承担隔离责任。

### 3.3 导出与异步任务隔离

- 导出、批量导入、长耗时异步任务也必须沿用同一租户口径。
- 不允许因为进入后台任务就绕开租户校验。

## 4. 当前落地状态

- 表结构与全局配置已经体现 `tenant_id`。
- 但当前 `WarehouseServiceImpl` 的方法体中尚未看到显式租户过滤或越权错误语义。
- 因此本规则页是必须遵守的项目级规则，同时也是当前实现缺口的检查单。

## 5. 禁止事项

- 前端直接决定最终租户范围
- Mapper/SQL 忽略租户上下文直接扫全表
- 删除、出库、导入时不做租户归属校验

## 6. 验收标准

- 查询接口不能读到其他租户数据
- 写操作不能修改其他租户数据
- 越权时返回稳定、可读的错误语义
- 相关 smoke 必须覆盖查询越权和写越权两类场景
