## 1. 目标

## 5. 建议补齐顺序

1. 补权威 PRD / 设计源锚点
2. 补 feature 级规格真源与覆盖关系说明
3. 冻结前端正式技术栈与后台 UI 方案
4. 补 RACI 与发布负责人
5. 补删除守卫、多租户、权限边界等高风险规则的落地证据

## 6. 使用规则

- 只要缺口会影响实现或验收，就优先记录在这里。
- 未补齐的事项可以先在项目级文档中写“待补”，但不能伪造结论。
- 缺口关闭后，要同步更新对应知识页，不只改本台账。
