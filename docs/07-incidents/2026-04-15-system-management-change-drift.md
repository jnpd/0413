# 2026-04-15 系统管理 change 规格跑偏复盘

## 1. 摘要

本次问题不是项目级规则互相冲突，而是写 change 时选错了规格真源，并且没有把基础 change 与完整业务 spec 分开。

直接表现为：

- 企业管理被误写成占位，遗漏新增 / 删除等真实能力
- 人员管理里的“企业管理员角色自动解析”没有写清，容易让人误以为是页面自由选择
- 角色管理只写到了授权树读取，没有把“保存授权 / 配置权限”闭环写完整

## 1.1 一眼看懂

| 问题 | 原因 | 修正 | 下一次怎么避免 |
|---|---|---|---|
| 企业管理被写成占位页，漏了新增 / 删除 | 把“没搜到代码”误判成“能力不存在” | 把企业管理拉回模块 PRD / 数据库设计的真实口径，明确它是有查询、保存、删除的真实页面 | 写 change 前先定真源顺序；没搜到代码时只能写“待校准”，不能写“不存在” |
| 人员管理看起来像“被固定选择” | 没把“企业管理员角色按企业类型自动解析”写清楚 | 把这条规则明确写成后端硬规则，不再让人误会成单纯的页面限制 | 以后把“页面可选项”和“后端硬规则”分开写，尤其是角色、状态、权限 |
| 角色管理像是少了“配置权限” | 只写了读授权树，没有把保存授权写进闭环 | 把“读取授权树 + 保存授权 + 权限反馈”视为同一能力，不再只写半截 | 以后凡是授权、删除、保存这类动作，默认检查“读 + 写 + 反馈 + 越权处理”四件事 |

## 1.2 第一性原理

从第一性原理看，这类问题本质上不是“AI 文笔不好”，而是下面三件基础判断错了：

1. 规格文档的职责，不是把眼前搜到的代码抄一遍，而是声明“系统里什么是真的”
2. “暂时缺证据”和“证据证明不存在”是两回事，不能混为一谈
3. 后台管理模块不是单页面问题，而是“数据实体 + 权限边界 + 读写闭环 + 跨模块联动”的组合

所以只要第一步真源选错，后面再认真写，也只是把错误写得更完整。

## 1.3 最终核对结果

这次重新按“PRD -> 后端 -> 前端”核对后，结论需要明确修正：

| 核对项 | 最终结论 | 依据 |
|---|---|---|
| 企业管理新增 / 编辑 / 删除是否在 PRD 中存在 | 存在 | [`模块PRD.md`](/E:/demo0413/docs/04-modules/01-系统管理模块/模块PRD.md) 已明确写有“新增企业、编辑企业、删除企业” |
| 企业管理新增 / 编辑 / 删除是否在后端中存在 | 存在 | [`EnterpriseController.java`](/E:/demo0413/bladex-boot/bladex-boot/src/main/java/org/springblade/modules/system/controller/EnterpriseController.java) 已提供 `/page`、`/submit`、`/remove` |
| 企业管理新增 / 编辑 / 删除是否在前端中存在 | 存在 | [`EnterpriseManagementTab.tsx`](/E:/demo0413/frontend/src/features/system/enterprise/EnterpriseManagementTab.tsx) 已有新增弹窗、编辑弹窗、删除动作 |
| 人员管理“被固定选择”是不是代码缺陷 | 不是单纯缺陷，更准确地说是业务规则没有在旧 spec 里写清楚 | 企业管理员角色本来就应按企业类型自动解析，不是任意下拉自由选 |
| 角色管理是否缺少“配置权限”能力 | 当前代码里不缺，缺的是旧 spec 没把“保存授权”闭环写完整 | 当前前端已有授权弹窗和保存授权动作 |

因此，本次真正要修正的不是“企业代码没有新增 / 修改 / 删除”，而是“旧 change / spec 对现有能力的描述错误”。

## 2. 现象

### 2.1 企业管理

- 文档里把 enterprise 当成“后端缺口占位页”
- 结果漏掉了新增企业、删除企业、删除守卫、平台超管按钮可见性等能力

### 2.2 人员管理

- 文档没有把“企业管理员角色按企业类型自动解析”写成硬规则
- 结果阅读者会把页面理解成“人员类型和所属角色都可自由选”

### 2.3 角色管理

- 文档写到了读取授权树
- 但没有把“保存授权 / 配置权限”当成同一闭环的一部分

## 3. 根因

### 3.1 真源判断错误

把“当前没搜到实现锚点”误判成了“该能力不存在”。

正确判断应该是：

- 如果模块 PRD、数据库设计、测试用例已经冻结能力，则业务真相成立
- 没搜到代码时，只能写“实现锚点待校准”

### 3.2 把基础 change 写成了完整业务 spec

基础 change 适合冻结：

- 登录
- 路由
- 菜单
- 页面拆分

但不适合在没有完整交叉验证前，替代系统管理模块的完整业务真相。

### 3.3 没有写覆盖关系

后续纠偏 change 出现后，如果不写：

- 谁覆盖谁
- 覆盖哪一部分
- 旧 change 还剩什么范围

后面的 agent 很容易继续引用旧口径。

### 3.4 没做跨模块闭环

企业、人员、角色在这个域里不是孤立页面。

只写单页，不写联动，就会漏掉：

- 企业管理员关系
- 企业类型到管理员角色的映射
- 删除守卫对人员有效状态的依赖
- 角色授权的保存闭环

## 4. 裁决结论

本次不是这些项目级规则互相冲突：

- 表单提交契约
- 权限边界规则
- 多租户隔离规则

真正的冲突在于：

- 基础 change 的临时判断
- 对比
- 模块 PRD / 数据库设计 / 当前实现事实

也就是说，问题出在“真源选错 + 范围声明不清 + 覆盖关系缺失”，不是规则本身冲突。

## 4.2 旧 spec 不完整或已过时的地方

| 主题 | 旧口径 | 现在更准确的口径 |
|---|---|---|
| 角色列表接口 | 旧 spec 把角色管理主读取口径写成 `/blade-system/role/list` | 当前 PRD 与前端主列表口径已是 `/blade-system/role/page`，`/list` 不应再作为主列表真源 |
| 角色权限配置 | 旧测试口径容易让人理解成“菜单树、数据权限树、接口权限树都已在前端完整展示” | 当前前端只接菜单权限树，数据权限树和 API 权限树仍未在页面暴露 |
| 人员管理角色选择 | 旧文档没有清楚区分“页面可选项”和“后端硬规则” | 企业管理员角色应按企业类型自动解析，这不是普通自由选择 |
| 角色管理配置权限 | 旧 spec 写到了读取授权树，但没有把“保存授权”写成同一闭环能力 | 当前代码中已有“打开授权弹窗 -> 勾选菜单 -> 保存授权”的完整动作 |

## 4.3 两个容易误判的点

### A. 人员管理“被固定选择”到底是不是代码缺陷

结论要拆成两层：

| 层面 | 结论 | 说明 |
|---|---|---|
| 企业管理员角色自动解析 | 这是业务规则，不是缺陷 | 当前前端在选择“企业管理员”后会自动按企业类型解析角色，并禁用角色下拉；后端也会再次按企业类型解析角色，不接受任意自由输入 |
| 页面可理解性 | 这里仍可能有 UX / 文档问题 | 当前页面虽然实现了自动解析，但没有把“为什么被锁定、将自动分配什么角色”解释得很清楚，阅读旧 spec 时更容易被误会成“页面写死” |

进一步说：

- 前端证据：
  - `resolveAdminRoleId(...)` 会把企业类型映射到 `water_company_admin` 或 `water_factory_admin`
  - 当 `isAdmin === '1'` 时，`所属角色` 下拉会被禁用
- 后端证据：
  - `UserServiceImpl#resolveWaterUserRoleId(...)` 会再次按企业类型把管理员角色解析成固定 alias
  - 如果找不到对应管理员角色，会直接报错，而不是接受任意角色

因此，“企业管理员角色不能自由选”是明确业务规则。

但还要补一句更细的判断：

- 如果讨论的是“普通人员角色列表是不是只展示企业自定义角色”，当前实现未必能仅靠前端代码证明完全满足
- 因为前端是直接调用 `/blade-system/role/page`
- 当前后端 `RoleController#page` 能确认的是：对非管理员用户会排除 `administrator`、`admin`、`water_company_admin`、`water_factory_admin`
- 但这不等于已经严格证明“只剩企业自定义角色”

所以这块最准确的说法不是“完全没问题”，而是：

- 企业管理员角色自动解析：这是规则，不是缺陷
- 普通人员角色列表口径：还需要继续校验是否与“仅企业自定义角色”完全一致

### B. 角色管理是不是缺少“配置权限”能力

结论也要拆成两层：

| 层面 | 结论 | 说明 |
|---|---|---|
| 菜单权限配置 | 代码里已经有 | 前端已有授权弹窗、读取授权树、读取已授权菜单、保存授权动作 |
| 完整权限配置 | 现在还不完整 | 后端接口支持菜单权限、数据权限、API 权限，但当前前端只暴露了菜单权限树 |

当前代码的真实状态是：

- 有“配置权限”这件事
- 但这件事当前只做到“菜单权限配置”
- 还没有做到“完整权限配置”

直接证据：

- 前端会打开授权弹窗，加载 `grant-tree` 和 `role-tree-keys`
- 前端点击保存时会调用 `POST /blade-system/role/grant`
- 但构造 payload 时，`dataScopeIds` 和 `apiScopeIds` 直接传空数组
- 页面文案也明确写着“当前仅接入菜单权限树；数据权限树与 API 权限树保持空集合提交”

所以最准确的表述应该是：

- 如果你说“角色管理完全没有配置权限”，这不对
- 如果你说“角色管理还没有完整做完配置权限”，这成立

- user 的管理员角色解释口径
- role 的权限配置闭环口径
- 后续 AI 对系统管理 change 的阅读顺序

## 7. 下次应该怎么写

下次写类似 change，至少遵守下面四条：

1. 先确认真源顺序：模块文档 > 纠偏 change > 已验证实现 > 项目级规则 > 推断
2. 如果只是基础拆分 change，必须显式写“本 change 不冻结完整业务真相”
3. 如果是纠偏 change，必须显式写覆盖关系和阅读顺序
4. 只要涉及查询、保存、删除、授权、权限、租户、联动，就按闭环写，不能只写读取链路
